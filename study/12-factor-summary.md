# 12 Factor App 요약

## I. 코드베이스
- 1개 코드베이스 → 여러 배포(서버 100대든 1000대든 같은 코드)
- 서로 다른 앱이 코드를 공유하면 라이브러리로 빼서 관리

## II. 종속성
- 모든 종속성을 명시적으로 선언 (requirements.txt, Dockerfile)
- 시스템 도구(curl, ffmpeg 등)도 암묵적으로 의존하지 말고 Docker로 환경 자체를 패키징
- 서버 100대 띄워도 동일한 환경 보장이 핵심

## III. 설정
- DB 주소, API 키 등 배포마다 달라지는 값은 코드에 넣지 말고 환경 변수로 분리
- 설정 파일(`.env`, `config.yaml`)도 권장하지 않음 — 흩어지고 관리 어려움
- 실전에서는 AWS, K8s가 서버별로 환경 변수를 주입해줌 (코드에도 파일에도 없음)

## IV. 백엔드 서비스
- DB, 메시지큐, SMTP 등은 모두 "연결된 리소스"로 취급
- `os.environ["DATABASE_URL"]`처럼 환경 변수로 접근 → 코드 수정 없이 로컬 DB ↔ AWS RDS 전환 가능

## V. 빌드, 릴리즈, 실행
- 빌드(종속성 설치, 컴파일) → 릴리즈(빌드 + 설정) → 실행(그냥 띄우기)
- 빌드는 복잡해도 OK, 실행은 단순해야 함 — 실행 중 pip install 같은 거 하면 서버마다 버전 달라짐
- 각 릴리즈는 유니크한 ID로 관리

## VI. 프로세스
- 앱은 stateless 프로세스로 실행 — 서버 메모리에 데이터 저장 X, 외부 DB에 저장
- 로드밸런서가 아무 서버로나 요청 보내도 동작해야 오토스케일링 가능

## VII. 포트 바인딩
- 앱 자체가 포트를 열고 HTTP 요청을 받음 (외부 웹서버에 의존 X)
- uvicorn, Tornado, Jetty 같은 웹서버 라이브러리를 앱에 내장
- 실전에서는 앞단에 Nginx(리버스 프록시)를 두기도 함

## VIII. 동시성 (프로세스 타입으로 스케일링)
- 앱을 역할별 프로세스 타입으로 나눔 (web, worker, scheduler 등)
- 각 타입을 독립적으로 스케일링 — web 10대, worker 3대 식으로
- 앱은 절대 데몬화/PID 파일 관리 하지 않음 → 외부 프로세스 관리자에 맡김

| 방식 | 설명 |
|---|---|
| 데몬 + PID (X) | 앱이 스스로 백그라운드 실행, PID 파일로 자기 관리 → 중앙 통제 불가 |
| systemd (O) | OS 레벨 프로세스 관리자 — 단일 서버에서 재시작/상태 관리 |
| K8s (O) | 클라우드 분산 프로세스 관리자 — 서버 수백 대 스케일링, 자동 재시작, 오토스케일링 |

## IX. 폐기 가능 (Disposability)
- 그레이스풀 셧다운: 포트 닫기 → 처리 중인 요청 마저 끝냄 → 오래 걸리는 작업은 큐에 반환 → 종료
- 서버는 언제든 죽을 수 있다고 가정 — 오토스케일링 축소, 배포, 크래시 대응
- 견고한 큐잉 백엔드(RabbitMQ 등) 사용 권장 → worker가 죽어도 작업 유실 없음

## X. Dev/Prod 일치
- local/staging/prod 모두 같은 종류, 같은 버전의 백엔드 서비스 사용
- "로컬은 SQLite, prod는 PostgreSQL" 이런 거 하지 말 것
- Docker Compose로 로컬에서도 prod과 동일한 서비스(DB, 큐 등) 띄우기
- staging 환경은 prod 인프라를 Terraform(IaC)으로 축소 복제하여 구축

## XI. 로그
- 앱은 로그를 stdout에 출력만 함 — 파일 저장, 전송은 앱이 관여 X
- `logging` 라이브러리로 레벨(DEBUG, INFO, ERROR) 나눠서 stdout에 출력 → 로그 수집 시스템에서 필터링/분석 용이
- 로그를 어디로 보낼지(터미널, 파일, Datadog 등)는 실행 환경이 결정

| 환경 | 로그 흐름 |
|---|---|
| 로컬 | stdout → 터미널에서 실시간 확인 또는 파일 저장 |
| prod | stdout → Docker가 수집 → Datadog/ELK/CloudWatch 등으로 전송 |

## XII. Admin 프로세스
- DB 마이그레이션, 데이터 수정 스크립트 등 일회성 관리 작업
- 앱과 같은 환경, 같은 코드베이스, 같은 종속성에서 실행
- admin 스크립트도 앱 저장소에 커밋하여 관리 — 별도 분리 X
